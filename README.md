## ما هو هذا المشروع؟

هذا المشروع ليس بوت Telegram واحد — بل هو **محرّك توليد** (Generation Engine)
يأخذ وصفًا باللغة الطبيعية ويُنتج مشروع بوت Telegram كامل جاهز للتشغيل.

النظام مبني بطريقة معيارية (modular) حيث كل وحدة لها مسؤولية واحدة واضحة،
وجميع الوحدات متصلة عبر مخطط موحّد (Blueprint) ينتقل من مرحلة إلى أخرى.

## البنية المعمارية

```
telegram_bot_engine/
├── configuration/          # التكوين والإعدادات الافتراضية
├── logging/               # نظام التسجيل
├── blueprint/             # المخطط الأساسي
├── core/                  # العقود، السياق، الأخطاء، النتائج، الإقلاع
├── registry/              # سجل المكوّنات (Registry)
├── manager/               # مدير المحرّكات (Engine Manager)
├── engines/
│   ├── base/              # الفئة الأساسية للمحرّكات
│   └── generators/
│       ├── analyzer/      # محرّك التحليل (Spec 002) — 10 مراحل
│       └── project_planner/  # محرّك التخطيط (Spec 004) — 8 مراحل
├── builders/              # مُنشئات الملفات والأدلة
├── validators/            # المُدقّقات
├── pipeline/              # خط الإنتاج (6 مراحل + مُنسّق)
└── output/                # مدير المخرجات
```

## كيف يعمل النظام

1. **المستخدم** يكتب وصفًا باللغة الطبيعية (مثل: "أريد بوت متجر يدير المنتجات والطلبات باستخدام SQLite")
2. **محرّك التحليل** (Spec 002) يحلل الطلب عبر 10 مراحل وينتج `AnalysisReport`
3. **محرّك التخطيط** (Spec 004) يأخذ `AnalysisReport` (فقط — لا يقرأ الطلب الأصلي) ويُنتج `ProjectBlueprint`
4. محرّكات لاحقة (مستقبلية) ستولّد الكود الفعلي بناءً على المخطط

كل محرّك يقرأ فقط من السياق (Context) ما يحتاجه، وينتج ما عليه إنتاجه.
لا يحق لأي محرّك تعديل مخرجات محرّك آخر — التعديل يتطلب محرّكًا مخصصًا.
